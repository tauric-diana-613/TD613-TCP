## 0. Executive determination

Ash Keep presently contains a strong Flow-Core pedagogical field embedded inside a surrounding interface whose organization remains substantially pre-Flow-Core. The result is not a collection of isolated cosmetic defects. It is an architectural mismatch:

> Flow-Core has been mounted as a pedagogical island inside a legacy Ash interface that still organizes work through ontology-heavy forms, hidden rooms, internal nomenclature, coordinate-based scrolling, controls whose visible promise exceeds their state authority, and page structures that expose downstream obligations before the entrant can experience the relations those obligations govern.

The first Ash Flow-Core integration was intentionally scoped as a pedagogical façade over existing Ash state transitions. That scope protected custody semantics, hash behavior, lifecycle gates, release authority, and the zero-serverless-function budget. It also meant that the consequence grammar reached the ingress field before it reached the whole instrument.

This plan therefore governs a **whole-instrument pedagogical recompilation**, not a reskin, polish pass, copy edit, or decorative extension.

The recompilation must preserve:

- Ash custody, ancestry, local source anchoring, derivative eligibility, and continuity authority;
- Flow-Core as a deterministic pedagogical compiler and recommendation layer rather than a sovereign station;
- the canonical phase cycle `NOTICE → ACT → WORLD ANSWERS → NAME → REST → TRANSFER`;
- the four explicit, non-equivalent AIA routes;
- one canonical animation coordinator and one active field clock;
- progressive scalar descent from plain consequence to technical inspection;
- human closure and route selection;
- missingness, contradiction, abstention, and claim ceilings;
- separate authority at destination handoff;
- zero raw-content transport through the pedagogy layer;
- zero automatic release or custody action;
- full reduced-motion and static truth parity.

The recompilation must remove or repair:

- first-paint legacy flashes and visible cache-epoch residue;
- arbitrary viewport transport;
- dead, hoverable, or empty affordances;
- empty drawers and blank quarter-page panels;
- internal architecture labels presented as primary human language;
- form-first Room, Route, Map, Capsule, Draft, Hush, and Save Point surfaces;
- a fixed consequence animation reused without local workspace meaning;
- menu rooms that duplicate, orphan, or conceal primary workflow;
- demo hydration fragmentation;
- route controls that visually activate without visibly recompiling the page;
- interface states whose visual answer cannot be connected to the action that produced them.

## 2. Mathematical route-burden program

The whole-instrument recompilation shall use the existing Flow-Core route-burden surrogate as a design and validation instrument:

\[
B_{\gamma,t}=
\int_\gamma
\frac{Q_{\downarrow}(s,t)}
{P_{\mathrm{leg}}(s,t)A_{\mathrm{aff}}(s,t)}\,ds
+
\alpha\sum_{(r,s)\in\gamma}\delta_{rs}(t)
+
\beta\lVert H_\gamma(t)\rVert
+
\zeta N_{\partial W}(\gamma,t)
\]

where:

- `Q_downstream` is unresolved downstream dependency;
- `P_leg` is explanatory permeability;
- `A_aff` is meaningful affordance cross-section;
- `delta_rs` records seams between local interface sections;
- `H_gamma` records route-history deformation;
- `N_boundary` counts consequential boundary crossings.

The red-pen findings map directly to the surrogate:

| Observed interface condition | Route-burden effect |
|---|---|
| Hoverable but dead control | `A_aff → 0` while visual salience remains high. |
| Internal terminology before experienced relation | `P_leg ↓`. |
| Empty drawer or blank panel | `delta_rs ↑`; the control and promised state fail to glue. |
| Important workflow hidden in a command-only room | `N_boundary ↑`. |
| Navigation that changes tab but lands at an arbitrary shared coordinate | `||H_gamma|| ↑`. |
| One fixed animation repeated across non-equivalent workspaces | local anisotropy collapses toward isotropic repetition. |
| Forms exposing all future obligations at once | `Q_downstream ↑`. |
| Dead Rest, Previous, or Next controls | recovery and closure cross-section collapse. |

### 2.1 Operational design objective

For each principal route through Ash Keep, the implementation should reduce burden by:

\[
Q_{\downarrow}\downarrow,
\qquad
P_{\mathrm{leg}}\uparrow,
\qquad
A_{\mathrm{aff}}\uparrow,
\qquad
\delta_{rs}\downarrow,
\qquad
\lVert H_\gamma\rVert\downarrow,
\qquad
N_{\partial W}\downarrow
\]

without flattening legitimate authority boundaries or making non-equivalent routes falsely interchangeable.

### 2.2 Route-burden calibration posture

No universal cognitive threshold shall be declared. Initial thresholds remain local and experimental. The implementation may compare pre- and post-recompilation observations, but any claim that burden has decreased must be tied to observable interface behavior and task evidence.

## 4. Program architecture

The implementation shall introduce one whole-instrument pedagogy composition layer with five cooperating responsibilities:

1. **Scene compiler** — compiles the local Ash workspace, profile, lifecycle state, route posture, missingness, and available actions into a bounded `PedagogicalScene` and `PedagogicalTransition`.
2. **AIA view compiler** — compiles one of four explicitly selected non-equivalent views while preserving shared invariants.
3. **Workspace renderer** — renders child-legible panels, stateful controls, static truth, local consequence, and inspection descent.
4. **Navigation covenant** — transports intention through workspace, heading focus, meaningful scroll destination, visible page confirmation, and return receipt.
5. **Empirical witness layer** — records observable interface behavior and bounded study evidence without learner-state inference.

The architecture shall preserve one canonical consequence-field instance and one active animation clock. Workspace changes recompile the field scene; they do not spawn additional field instances.

## 7. Acceptance matrix

A release candidate fails when any principal journey contains:

- a visible dead control;
- an empty drawer;
- a route button that changes styling without visible recompilation;
- a workspace transition without meaningful heading and anchor;
- a background viewport reset after entrant-owned movement;
- duplicate field instances;
- a field animation unrelated to local workspace state;
- a technical label required before meaningful action;
- a hidden primary workflow reachable only through the command menu;
- a relation that cannot be found after creation;
- a Rest action that removes inspection or return;
- an unannounced boundary crossing;
- a generated explanation required for navigation;
- source bytes represented as transported when they remain local;
- contradiction or missingness visually smoothed away;
- a route inference made without explicit operator selection;
- a custody, release, or closure action performed by pedagogy code.
