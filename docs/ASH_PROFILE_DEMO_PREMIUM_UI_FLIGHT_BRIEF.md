## 2. Executive diagnosis

Ash currently looks like an unusually intelligent laboratory interface. It does not yet look like the indispensable premium operating instrument implied by the data underneath it.

The distinction is not a request for ornamental gloss.

An expensive product communicates, before the operator reads the manual:

```text
I know what kind of work this is.
I know what matters now.
I know what is safe to touch.
I know what has already left.
I know what can wait.
I know how to return tomorrow.
```

The present interface communicates its constitutional seriousness, but it still makes the operator perform too much visual sorting and translation. The data is wearing a lab coat when it needs a tailored travel suit.

## 3. What already works

### 3.1 The map has genuine product magnetism

The room geometry is the strongest visible signature in Ash. Campaign and fundraiser data produce meaningfully different spatial fields. The map does not read as a decorative network graph; it reveals that a campaign and a fundraiser organize risk differently.

The map should remain the visual center of gravity.

### 3.2 Profile identity survives into the constitutional object

`political_campaign` and `fundraiser` are real Case Map profile identities rather than labels pasted over `research` or `organizing`. This gives future dashboards, Choir assays, Capsules, Hush review, and destination handoff an honest occupational root.

### 3.3 Route Memory becomes intelligible with realistic data

Three prehydrated routes per demo make the difference between a blank receipt system and a visible history of actual crossings. The campaign route cards show reporter, coalition, and event-vendor boundaries. The fundraiser route cards show host, sponsor, and vendor boundaries.

### 3.4 The Draft chamber benefits immediately from specialization

Profile-specific recipient, purpose, references, protected literals, provider task, and tradeoff posture make Draft feel like work rather than a schema demonstration.

### 3.5 Mobile integrity is structurally sound

The flight found zero document-level horizontal overflow. Workspaces remain reachable and the dense map can render on a phone without collapsing the page.

That is a foundation, not the finished mobile experience.

## 4. Seams exposed by the hydrated data

### 4.1 Ash opens on a map when the operator first needs a command deck

A 32-object campaign and a 33-object fundraiser arrive as a complete topology. That is intellectually rich and operationally abrupt.

A person opening Ash between calls needs, first:

```text
what changed
what is urgent
what is held
what is awaiting approval
what leaves next
what must remain local
```

The map should be one tap away, but the first post-hydration surface should be a profile-aware **Command Deck**.

### 4.2 The global navigation exposes implementation organs rather than work intent

`Custody / Map / Rooms / Routes / Test / Draft / Save` describes Ash’s internal anatomy.

A premium pocket instrument should translate those organs into operator intentions while preserving the exact constitutional names beneath them.

Recommended primary mobile navigation:

```text
Home
Map
Work
Choir
Capsule
```

Recommended secondary sheet or command palette:

```text
Custody
Rooms
Routes
Draft and Hush
Destination Handoff
Receipts
Settings and profile
```

`Work` may present profile-specific queues while routing each action into its real constitutional workspace.

### 4.3 The lifecycle vocabulary is too raw for the primary visual hierarchy

`ARRIVAL_UNPERSISTED`, `CASE_BOUND`, and `REBUILD_ELIGIBLE` are important exact states. They should remain inspectable.

They should not be the dominant language confronting a busy operator.

Use paired language:

```text
Human posture: Setup not finished
Exact state: ARRIVAL_UNPERSISTED
Next action: Bind or verify custody
```

This preserves rigor while reducing translation labor.

### 4.4 The interface is visually flat despite having deep structure

The present field relies heavily on one dark plane, thin outlines, a persistent grid, all-caps microtype, and similarly weighted cards. This makes expensive data look inexpensive because almost every object occupies the same visual temperature.

Premium treatment should add hierarchy through material and light:

```text
smoked deep-green base
near-black elevated sheets
a warm ivory reading surface for bodies and receipts
restrained brass or old-gold authority accents
mint reserved for verified local action
rose reserved for holds and unresolved risk
violet reserved for emergent or Choir residue
soft depth shadows instead of borders around every object
subtle glass blur only on transient navigation and command surfaces
```

The grid should become atmospheric rather than omnipresent.

### 4.5 Serif and mono are doing too many jobs

The editorial serif gives Ash character. The mono gives receipts jurisdiction. The current interface often uses them everywhere, which weakens both.

Recommended typography roles:

```text
editorial serif = case identity, room names, major decisions
high-quality humanist sans = operational reading, controls, cards
mono = exact IDs, digests, states, route receipts, provenance
```

Long form inputs and operational labels need the sans. Machine-facing records keep the mono.

### 4.6 Mobile is technically intact but physically exhausting

The campaign mobile map screenshot becomes a very long column:

```text
header
seven-tab rail
case title
map
map controls
legend
posture
object registry
add form
connect form
```

Nothing clips, but the operator must walk through the whole building to turn off one lamp.

Premium mobile composition requires:

- a five-item bottom dock;
- sticky profile action bar;
- object details in a bottom sheet;
- add/connect forms in a single action drawer;
- map controls in a compact floating sheet;
- collapsed registries with count, filter, and search;
- an always-visible return to current priority;
- 44–48 px minimum touch targets;
- thumb-reachable primary actions;
- preservation of ordinary vertical scrolling without nesting several competing scroll fields.

### 4.7 Route Memory reads like storage, not operational history

Route cards currently lead with machine route IDs. Those IDs matter, but human history should lead with:

```text
recipient
purpose
when it left
how many references left
recall or receipt posture
```

The exact route ID may sit underneath.

Recommended route presentation:

> Reporter · Schedule inquiry  
> Three references left · 12 minutes ago  
> Receipt held locally · Not recalled  
> `route_reporter_response`

A chronological crossing timeline will feel both more premium and more useful.

### 4.8 Review is rigorous but visually undifferentiated

Thirteen checkboxes presented at equal weight look like a compliance form. The operator needs grouped review intelligence:

```text
Custody and integrity
People and confidentiality
Route and metadata
Reconstruction and Choir
Final exact-draft confirmation
```

Each group should show `clear`, `held`, or `needs review`, with individual checks available beneath.

### 4.9 Capsules exist, but they do not yet behave like a product promise

Capsule export and recovery are buried under Save. The blank-browser recovery door is now present, but continuity should be visible throughout Ash.

Recommended Capsule product object:

```text
Last sealed
Current case protected / not protected
Release receipt bound / absent
Passphrase never stored
Export encrypted copy
Open encrypted copy
Return to last Save Point
```

Capsule should be a primary destination and a persistent small continuity indicator. It is Ash’s pocketability made literal.

### 4.10 Choir remains the most consequential missing product organ

The hydrated fixtures now provide the cross-Room density Choir needs. They expose why a generic Rebuild Test cannot substitute for the missing Choir interface.

A dedicated Choir workspace should eventually present:

```text
Pairwise Moiré
Calibration circuit
Reader provenance
Reader disagreement
Matched benign controls
Higher-order interference
Ordered route sequence
Temporal and delayed disclosure
Receipt and replay
```

The surface should visualize interference residue without converting it into causation, identity, truth, prediction, suppression, or action authority.

Recommended Choir visual grammar:

- projection cards as selectable voices or parts;
- a combination matrix for pairwise assays;
- a bounded lattice for higher-order assays;
- a route strip for ordered-sequence controls;
- a temporal score for immediate, delayed, reset, and cross-session observations;
- violet residue layered over the existing Room map;
- explicit matched-control and missing-evidence lanes;
- one persistent claim-ceiling drawer.

Choir should feel like the cathedral instrument inside Ash—not another receipt textarea.

## 6. Premium interaction law

Looking expensive cannot come at the cost of evidentiary honesty.

Every premium interaction should preserve:

```text
visible source status
visible hold reason
exact before/after state
operator gesture
what left / what remained
undo or successor posture
receipt access
no silent mutation
```

Recommended motion:

- 120–180 ms state transitions;
- subtle spatial continuity between Home, Map, and object sheet;
- no decorative continuous animation in work views;
- reduced-motion parity;
- verified-action glow that resolves rather than pulses forever;
- local save acknowledgment within one frame, followed by exact receipt completion.

Recommended feedback:

- tactile-looking pressed states;
- optional platform haptics where available;
- meaningful skeletons rather than blank panes;
- no spinner without a named operation;
- every hold explains the next lawful action.

## 7. Reconciliation sequence

The UI should not be polished in one giant visual rewrite. The correct sequence is:

### Phase A — Product inventory

Reconcile all eleven-Stretch organs as:

```text
integrated operator UI
standalone discoverable UI
engine-only
receipt-only
test-only
absent
```

This includes Choir, Capsules, Safe Harbor ingress, provenance adapters, destination handoff, Hush interventions, Custodian Return, and Cinder boundaries.

### Phase B — Premium shell

Build:

- profile-aware Command Deck;
- five-item mobile dock;
- human/exact paired lifecycle language;
- premium material and type system;
- drawers and sheets for dense operations;
- search and command palette;
- persistent Capsule posture.

### Phase C — Specialized dashboards

Compose the Political Campaign and Fundraiser decks from the same constitutional objects rather than separate mini-apps.

### Phase D — Choir arrival

Construct the dedicated Choir workspace and bind it to current Case Map, Route Memory, Reader registry, calibration, higher-order, sequence, and temporal receipts.

### Phase E — Integrated crossings

Make Safe Harbor ingress and destination handoff discoverable within the product while preserving their exact gates and claim ceilings.

### Phase F — Next occupational fixture

Only after Campaign and Fundraiser flows are refined should the Real Estate Developer profile be authored. Its fixture should then stress capital stack, entitlement, site control, design, contractors, public agencies, community commitments, schedule, documents, external routes, and unresolved approvals without pretending to supply legal, financial, engineering, or market truth.

## 9. Product ruling

The profile hydration strategy was correct.

Without realistic Campaign and Fundraiser data, a visual redesign would have optimized the empty shell. The new fixtures reveal the actual product requirement: Ash must become a **profile-aware command instrument whose constitutional machinery stays exact while its operational face becomes immediate, tactile, and calm**.

The next UI packet should not merely make the current page prettier. It should reconcile the missing organs, introduce the Command Deck, establish the premium mobile shell, and prepare a first-class Choir and Capsule experience.

No score, maturity, production class, transport authority, release authority, identity claim, truth claim, suppression authority, or Cinder authority is changed by this brief.
