# Human Guide

𝌋‌ TCP should read like one instrument with multiple human consequences, not like three unrelated products taped together.

This guide is for a shared audience:

- a whistleblower, witness, or high-stakes reader trying not to overclaim
- a person exploring cadence because language behavior matters
- an RLHF or post-training reader looking for explicit contracts and proof surfaces
- anyone else curious enough to test the instrument carefully

## What anyone should understand first

TCP is a bounded stylometric and routing instrument. It compares patterned language, stages shell borrowing, and names route conditions like weak signal, branch hold, criticality, and passage. It does not grant magical certainty.

The most important discipline in the whole project is this:

- pattern is not verdict
- recognition is not route
- residue is not noise just because it is unresolved

That single discipline matters to all three readers named above.

## What the app is trying to make visible

TCP is built to expose four kinds of state without blurring them together:

- weak resemblance that may be noise
- real cadence pressure that still lacks explanation
- residue that should be preserved rather than flattened
- routing conditions that may lower burden without breaking provenance

If the app is doing its job, a human reader should be able to tell which of those conditions they are looking at.

## How to use it well

The cleanest shared workflow is:

1. keep the raw text intact in the bays
2. press `Analyze Cadences`
3. read the shell cards before interpreting the duel
4. press `Swap Cadences`
5. compare raw text, duel output, and retrieval-facing summaries together
6. only then decide whether you are seeing weak signal, branch pressure, or a meaningful borrowed-shell event

That procedure is good for a cautious witness, a curious explorer, and a training team alike.

## If the stakes are human

If you are reading TCP in a whistleblower or witness context, the app can help you keep a pattern visible without forcing it into a false claim too early.

What it can do:

- preserve comparison
- surface residue
- show that a patterned relation is not random noise
- help distinguish blocked movement from real donor-shaped movement

What it should not be asked to do alone:

- prove authorship
- prove motive
- prove coordination
- replace provenance, testimony, or documentary context

The safest phrasing is usually not “TCP proved this.”  
The safer phrasing is “TCP surfaced patterned pressure worth preserving and routing, and here is the rest of the evidentiary context.”

## If the curiosity is about cadence itself

TCP is also a playful instrument. It is allowed to be one.

You can use it to ask:

- what changes when only the shell changes
- which voices lend structure cleanly and which ones stall
- how contraction posture, connector stance, abstraction, and lexical framing move together
- when a persona becomes reusable

The best exploratory sequence is:

1. analyze the seeded pair
2. swap cadences
3. randomize or replace one side
4. save a persona
5. open `Trainer`
6. build a persona from a longer corpus
7. validate pasted model output against the retrieval lane

That path keeps the play connected to the proof surface instead of drifting into pure vibes.
